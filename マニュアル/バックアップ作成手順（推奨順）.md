以下は、**TUFF-OS** における**バックアップ作成手順**の詳細です。  
この手順は、以下の目的で使用してください：

- Recovery PIN紛失・再インストール前の予防
- 物理ディスク全損・交換前の保険
- 定期メンテナンス時のデータ保全
- テスト環境から本番への移行

**重要警告（必ず最初に読んでください）**

- バックアップ中は**電源を絶対に切らない**でください（電断時は自動ロールバック機能が働く設計ですが、念のためUPS推奨）。
- バックアップ先は**TUFF-OS外の安全なストレージ**（外部HDD、NAS、暗号化クラウドなど）を使用。
- バックアップファイルは**暗号化**（GPGまたはVeraCrypt推奨）して保存。
- バックアップ取得後、**必ず整合性検証**を行ってください。

### バックアップ作成手順（推奨順）

#### 方法A: 論理バックアップ（最も推奨・高速・柔軟）

**対象**: 重要なフォルダ（/data、/homeなど）  
**メリット**: 容量が少なく、リストアが簡単  
**所要時間**: 数分〜数十分（データ量による）

1. **ログイン状態で実行**  
   ```bash
   tuffutl sys login <your_id> --password <your_pw>
   ```

2. **バックアップコマンド実行**（推奨オプション付き）
   ```bash
   tuffutl backup create \
     --target /data \                  # バックアップ対象ルート（複数指定可）
     --output /mnt/external/backup_$(date +%Y%m%d).tar.zst \
     --compress zstd \                 # 高圧縮（zstd推奨）
     --encrypt --key /path/to/keyfile \ # 暗号化（任意だが強く推奨）
     --verify \                        # 作成後に自動検証
     --exclude "/data/temp"            # 不要ディレクトリ除外（任意）
   ```

   - **主なオプション解説**
     - `--target`：バックアップ対象パス（複数指定可：`--target /data --target /home`）
     - `--output`：出力先ファイル（.tar.zst推奨）
     - `--encrypt`：GPG暗号化（PINまたは公開鍵指定）
     - `--verify`：作成後に自動で整合性チェック
     - `--dry-run`：事前シミュレーション（安全確認用）

3. **バックアップ完了確認**
   ```bash
   tuffutl backup verify --file /mnt/external/backup_20260322.tar.zst
   ```
   → 「Verification successful」と表示されればOK

4. **バックアップファイルを安全な場所へ移動**  
   - 外部HDD / NAS / 暗号化USBへコピー  
   - 複数箇所に分散保存（3-2-1ルール推奨：3コピー、2メディア、1オフサイト）

#### 方法B: 物理JBOD全体バックアップ（完全復元用）

**対象**: 全HDD内容を丸ごと保存  
**メリット**: 完全な状態復元が可能  
**デメリット**: 容量が非常に大きい（数TB以上）

1. **システムを停止**（推奨）
   ```bash
   tuffutl sys poweroff
   ```

2. **各HDDをddでイメージ化**（Linuxホスト側で実行）
   ```bash
   # 各HDDごとに実行（例: sdb, sdc, sdd）
   dd if=/dev/sdb of=/mnt/backup/jbod_sdb_$(date +%Y%m%d).img bs=4M status=progress conv=fsync
   dd if=/dev/sdc of=/mnt/backup/jbod_sdc_$(date +%Y%m%d).img bs=4M status=progress conv=fsync
   # ... 必要数繰り返し
   ```

3. **イメージファイルの整合性確認**
   ```bash
   sha256sum /mnt/backup/jbod_*.img > checksums.txt
   # 後日検証用にchecksums.txtも保存
   ```

4. **イメージファイルを安全な場所へ移動**  
   - 外部ストレージ複数箇所にコピー  
   - 暗号化必須（VeraCryptコンテナ推奨）

#### 方法C: 差分バックアップ（定期運用向け）

**対象**: 前回バックアップからの変更分のみ  
**メリット**: 容量節約・高速

```bash
tuffutl backup incremental \
  --base /mnt/backup/full_20260301.tar.zst \
  --target /data \
  --output /mnt/backup/incremental_20260322.tar.zst \
  --verify
```

### バックアップ後の必須確認項目

1. **整合性検証**  
   ```bash
   tuffutl backup verify --file <backup_file>
   ```

2. **リストアテスト**（定期的に実施推奨）  
   - テスト用VMでリストア実行 → データ内容・TagGroupMask・権限が完全に復元されるか確認

3. **PINとバックアップの紐づけ**  
   - バックアップと同時にRecovery PINも再確認・記録  
   - PIN紛失時はバックアップがあっても**完全復旧不可**（再インストール必須）

### バックアップ運用ベストプラクティス

- **3-2-1ルール厳守**  
  3コピー、2種類のメディア、1つはオフサイト
- **定期実行**  
  週1回フルバックアップ、毎日差分バックアップ
- **暗号化必須**  
  バックアップファイルは必ずGPGまたはVeraCryptで暗号化
- **テスト復元**  
  3ヶ月に1回はリストアテストを実施
- **PIN管理**  
  PINはバックアップと一緒に**別メディア**に保存（紙＋金庫推奨）

**まとめ**  
バックアップはTUFF-OSの「**最後の保険**」です。  
PIN紛失や物理全損時でも、バックアップがあれば**最小限のダウンタイムで復旧**可能です。  
しかし、**バックアップ自体を失うと完全喪失**となります。  
**バックアップ管理はシステム運用の中で最優先事項**としてください。
